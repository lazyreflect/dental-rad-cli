**Author:** research-session Claude (cold start)

**Status:** Research only — no code written. Findings inform v0.5 scoping.
**Trigger:** Renielaz `dental-caries-x-ray` blocker (class list contaminated by 6 description-bullet "classes"; only 205/2978 annotations bound to real ICCMS classes; RC6 has 13 samples). Caries deferred from v0; bone-loss pipeline trains tonight.

## TL;DR

**Primary recommendation: Path D (Baasils ICCMS) — probe first, then train.**
The Baasils dataset (`baasils-workspace/iccms-dental-caries-etomb`) appears to be
the dataset behind the published Salehizeinabadi 2025 paper which reports

description-bullet corruption, and adequate per-class counts. Public Domain
license. Concrete next-action: 60-minute probe with `roboflow` SDK to confirm
class list + per-class annotation counts, then `caries_adapter.py` already
plugs in.

**Fallback if Baasils is also corrupted:** Path E + binary fallback —
hand-label ~150 of Joseph's own BWs using LabelStudio with active-learning
pre-annotation from a binary detector trained on DentalAI. Ships a binary
`caries_present: bool` finding for v0.5 and bootstraps a real ICCMS-tier
dataset over the next 4-6 weeks.

**Reject:** Path B (math doesn't change — 205 real annotations stay 205 after
4-8 hrs of cleanup; depth tier still starves). Path F (caries findings are
load-bearing for the chairside doc-aid use case — see §5).

## 2. Path B — Fix Renielaz upstream (REJECTED)

### What I confirmed
- Roboflow's **Modify Classes** preprocessing (rename/merge/delete classes at
  version-generation time, preserving annotations) is a **Roboflow Pro feature**.
  Joseph's free Public plan does not have it. Confirmed from the
  Class Management blog post and Roboflow docs.
- Project-level class deletion (free tier) keeps the bounding box / polygon
  geometry but reassigns the class to `null`. The 2,773 garbage-bound
  annotations would have to be **manually re-bound or re-deleted image-by-image**
  in the annotation UI.
- The 205 real ICCMS annotations stay 205. **Cleanup does not produce new data.**
- RC6 deep tier remains stuck at 13 samples — below the ≥30 floor.

### Time cost vs payoff
- Fork project: ~2 min.
- Walk 1,483 images to re-bind/delete garbage annotations: **6-10 hrs UI work**
  (estimate based on ~3-4 sec per annotation × 2,773 annotations + page-load
  overhead).
- Result: marginally cleaner Renielaz with the same RC6 collapse problem.

### Verdict
**Rejected.** Cleaning Renielaz does not change the underlying class
imbalance. Even with perfect labels, the deep tier won't generalize. Spending
8 hours to fix a structurally undersized dataset has poor ROI when Baasils
(§3) likely has the same lesions already cleanly labeled.

### Salvage value
If Path D fails, the cleaned Renielaz 205 annotations can be **merged** with
Joseph's hand-labeled BWs (Path E) to bootstrap the binary detector. Keep the
fork option open but don't burn cleanup hours unless Path D also fails.

## 3. Path D — Alternative public datasets

I probed every candidate the original survey named, plus four that it missed.
Findings, ordered by attractiveness:

### 3.1 ✅ **Baasils ICCMS** (`baasils-workspace/iccms-dental-caries-etomb`) — STRONGEST CANDIDATE

| Property               | Value                                                       |
|------------------------|-------------------------------------------------------------|
| Class list             | `Negative, Positive, RA1, RA2, RA3, RB4, RC5, RC6` (8 classes — clean) |
| Modality               | Bitewing                                                    |
| Images (latest seen)   | 537 base / 730 with augmentation (per Salehizeinabadi 2025) |
| Annotations            | 1,115 carious lesions                                       |
| License                | Public Domain (per Roboflow Universe listing)               |
| Last update            | "4 days ago" as of search date                              |

**Critical evidence it is NOT corrupted like Renielaz:**

The Salehizeinabadi et al. 2025 paper ("Automated Classification of Dental
Caries in Bitewing Radiographs Using Machine Learning and the ICCMS
Framework", *International Journal of Dentistry*, PMC12393923) trained
YOLOv11 against what is almost certainly this dataset (730 BWs / 1115
lesions matches; uses Roboflow platform). Reported per-class mAP50:

- RA1+RA2 (merged): 0.61
- RA3:               0.52
- RB4+RC5 (merged):  0.74
- RC6:               0.80

You cannot get mAP50 = 0.80 on RC6 with only 13 training examples. The fact
that the paper reports meaningful per-class metrics across all six ICCMS
tiers is **direct evidence the class distribution is real**, not the
description-bullet artifact we saw in Renielaz.

The class list `Negative, Positive, RA1...RC6` is also semantically sane —
"Positive" is the wrapping caries-present binary, "Negative" is the
no-caries flag, and the RA*/RB*/RC* are the depth tiers. That's an
intentional dual-level ontology, not a corruption.

**MUST-DO probe before committing** (the lesson from Renielaz): same probe
script that exposed Renielaz's bug. Concrete commands:

```bash
# Use the existing probe scripts, retargeted:
python scripts/_probe_roboflow_versions.py \
    --workspace baasils-workspace --project iccms-dental-caries-etomb
python scripts/_probe_roboflow_version_yaml.py \
    --workspace baasils-workspace --project iccms-dental-caries-etomb \
    --version <latest>
```

What to verify in the output:
1. `names:` field in `data.yaml` contains exactly the 8 expected class names
   (no markdown bullets, no "Roboflow is an end-to-end..." strings).
2. Per-class annotation count from sampled label files is roughly:
   `RC6 ≥ 30`, `RC5 ≥ 50`, `RB4 ≥ 80`, `RA3 ≥ 80`, `RA1+RA2 ≥ 200`.
   (Estimated by working backward from the paper's reported performance.)
3. Modality of sampled images is bitewing (cone-shaped molar/premolar view,
   not panoramic).

**Adapter changes:** trivial. `caries_adapter.py` already does a 3-class
collapse (RA1+RA2+RA3 → initial, RB4+RC5 → moderate, RC6 → deep). The only
modification is to **drop the Negative/Positive wrapper classes** during
adapter import. That's a 10-line patch.

**Time to v0.5 training kickoff if probe passes:** ~2 hours
(probe → adapter patch → download → run trainer).

**Risk:** Even if class list is clean, per-class counts might be tighter
than the paper implies (paper used augmentation to 2,190 images). If RC6
real-count is ~25-40, the 3-class collapse keeps `deep` separately
auditable; if it's <20, fall back to a 2-class collapse (`initial+moderate`
vs `deep`) or to binary.

**Verdict:** Probe first, expected to succeed, then ship.

### 3.2 ❌ AI-Dentify / HUNT4 — STRONGEST DATA, INACCESSIBLE

13,887 bitewings, expert-annotated by 6 dentists, labels merged into
`enamel caries`, `dentine caries`, `secondary caries`, `unknown`. Published
in BMC Oral Health (Pérez de Frutos et al. 2024).

**Why rejected:** Data is held by HUNT Research Centre (NTNU) and is
**not openly redistributed**. Per the AI-Dentify data availability statement,
"HUNT Research Centre minimises data storage outside its data bank and
refrains from depositing data in open repositories." De-identified data
require Regional Ethical Committee approval + research protocol approval.
Timeline: weeks to months. Conflicts with the ≤2-week constraint.

**Salvage value:** None for v0.5. Worth a contact-the-corresponding-author
email anyway for v1.0 horizon (their data dictionary is the gold standard
for proximal caries depth labeling). Joseph could put it on a Q3 2026 list.

### 3.3 ❌ Mendeley `4fbdxs7s7w` — TOO SMALL

100 bitewings, COCO bounding boxes by 8 annotators (5 experienced +
3 less experienced). Designed for **inter-annotator variability research**,
not training. License visible on page (likely CC-BY 4.0). 100 images is
well below the ≥500 floor; even as an external test set it's noisy because
the 8 annotators don't agree.

**Salvage value:** Useful as a **held-out evaluation set** once v0.5 trains.
Different annotator pool → measures generalization, not just held-out
performance on the training distribution.

### 3.4 ❌ DentalAI / DatasetNinja (`datasetninja.com/dentalai`) — WRONG ONTOLOGY

| Property      | Value                                          |
|---------------|------------------------------------------------|
| Images        | 2,495                                          |
| Annotations   | 28,904 labels                                  |
| Classes       | `Tooth, Caries, Cavity, Crack` (4 classes)     |
| Modality      | Mixed — appears to include both BW radiographs AND intraoral photos. Example `caries.jpg` in the Germanov repo is an intraoral photo. |
| License       | CC-BY 4.0 (per DatasetNinja listing)           |
| Format        | Supervisely → convertible to YOLO via Germanov's `convert.ipynb` |

Caries class is binary present/absent — **no depth tiers**. "Cavity" is
overlap with "Caries" but used inconsistently. Mixed modality is the bigger
red flag for our chairside-BW pipeline.

**Salvage value:** Binary-only fallback. If we filter to just BW images
(probably ~1,000-1,500) and use only the `Caries` class, this becomes a
CC-BY 4.0 ~1k BW binary caries dataset — strong floor for the
binary detector in §6.2. Same CLAHE preprocessing applies cleanly.

### 3.5 ❌ Salehizeinabadi 730-BW dataset — UNCLEAR PROVENANCE

The dataset in the paper IS Baasils (almost certainly — image count matches
perfectly; both used Roboflow), so this is the same line item as §3.1.
There's no separate download link in the paper supplementary. If §3.1 probe
fails, an email to corresponding author (Mehdi Salehizeinabadi, IJD 2025)
asking for the data export would take ~1 week round-trip.

### 3.6 ❌ Scientific Data 2025 intraoral dataset (`s41597-025-05647-9`)

6,313 intraoral PHOTOS (not radiographs), Mithi/Sindh Pakistan cohort,
ages 10-24, LabelMe annotations in YOLO/PASCAL/COCO formats. **Wrong
modality** — these are photographs of teeth, not bitewing X-rays.
Excellent dataset for photo-based caries detection but unusable here.

### 3.7 ❌ DENTEX challenge dataset (HuggingFace `ibrahimhamamci/DENTEX`)

Panoramic X-rays, MICCAI 2023 challenge. 4 diagnosis classes including
`caries` and `deep caries`. **Wrong modality** (panoramic, not bitewing).
Crowns/anatomy/scale all differ from chairside BW workflow.

### 3.8 ❌ Other Roboflow Universe datasets

- `code-geass/caries-classification-icdas-ii` (1,889 images, CC-BY 4.0):
  ICDAS-II ontology rather than ICCMS-radiographic, modality unclear from
  listings (likely mixed). Could be probed as a second fallback but lower
  priority than Baasils.
- `panoramic-xray-images/dental-caries-detection`: panoramic, reject.
- `dental-disease-segmentation/dental-caries-segmentation`: classes unclear,
  likely binary or panoramic.
- `jb-okcsk/kaggle-dental-caries`: Kaggle re-host, ontology unknown.

### 3.9 ❌ Other Mendeley datasets

- `9jnf2jvghy` Caries-Spectra: 2,000 intraoral photos, not BW.
- `c4hhrkxytw` Dental OPG: panoramic.
- `cgwnxmdp3b` Multi-Source: image-to-image translation source/target pairs,
  not annotated caries.
- `ctv68wsyv4` 3 most common diseases: classes/modality unclear.
- `wxv6h9p39g` 4 dental diseases: 4,000 OPG (panoramic) X-rays.
- `S2352340924009326` 6-class dental: 232/604 panoramic images.

**Net:** Mendeley caries datasets are either very small (100) or wrong
modality (OPG/photos). None viable as primary.

## 4. Path E — Hand-label Joseph's own BWs

### Realistic time estimate

Per LabelStudio docs and Roboflow annotation studies, an experienced
dentist annotating proximal caries with bounding boxes on a BW image:
- Empty/clean BW: 5-10 sec to confirm no lesions.
- BW with 1-3 lesions: 30-90 sec including class assignment (ICCMS tier).
- BW with 4+ lesions or interproximal ambiguity: 2-3 min.

At Joseph's reported volume (~30 BWs/day, ~70% will be normal), the
practical labeling throughput is **~20-30 BWs per hour** of dedicated
annotation time. To hit 500 labeled BWs, that's **17-25 hours** of
clinical-time labeling, spread over 4-6 weeks at ~3-5 hrs/week of nights
and weekends.

That's far longer than the active-learning bootstrap proposed in §6.2,
which gets to v0.5 in **3-4 hours** of labeling using a pre-trained binary
detector as the seed.

### Tool recommendation: LabelStudio (not Roboflow free tier)

| Factor                | Roboflow free | LabelStudio (local Docker) |
|-----------------------|---------------|----------------------------|
| Data privacy          | ❌ Public Universe — PHI scrubbing required AND license-compatible PHI exposure risk | ✅ Local-only, no upload |
| Annotation speed      | ✅ Good UI, polygon+bbox, hotkeys | ✅ Comparable; polygon, bbox, keypoint |
| ICCMS multi-tier      | ✅ Class hierarchy supported | ✅ Hierarchical labeling supported |
| Pre-annotation        | ✅ Roboflow Train + Label Assist | ✅ ML backend; easy to plug Joseph's YOLOv8 |
| Free tier credit limit| ⚠️ $60/mo credits, hits caps fast on training | ✅ Self-hosted = unlimited |
| Compounds locally     | ✅ via export | ✅ Native — annotations stay local |

**Recommendation: LabelStudio in Docker**, hosted on pickles (the RTX
4090 box). Joseph's PHI never leaves his network. Annotations export to
YOLO format directly via Label Studio's export config. Plug the bone-loss
model architecture for label-assist pre-annotation.

The PHI-scrub recipe in `docs/phi-scrub-recipe.md` still applies for the
**training images themselves** since YOLO training reads file paths and
the source DICOMs encode patient IDs in headers. Standard scrub:
DICOM → PNG with no DICOM metadata → randomized filename. Don't push the
scrubbed corpus to a public repo though — even scrubbed BWs of identifiable
patients require local-only storage for HIPAA hygiene.

### ICCMS calibration

Joseph's BW reading is calibrated to clinical practice, not necessarily
to ICCMS radiographic-scoring conventions. Before annotation kickoff he
should:
1. Read the ICCMS RA1-RC6 visual guide (figures in
   Salehizeinabadi 2025 supplementary, or the original ICCMS atlas).
2. Annotate ~20 calibration BWs, compare against the Baasils sample
   to gauge inter-rater agreement.
3. Lock the annotation policy (e.g. "anything ambiguous between RA3 and
   RB4 → annotate as RA3 with confidence flag"; or "skip ambiguous").

Total calibration time: **~1 hour**. Output: an `annotation-policy.md`
that Joseph can re-read between sessions to stay consistent.

### Compounding value

Per the customer-zero observation from work-root CLAUDE.md, an
office-specific labeled corpus is the asset that scales if `dental-rad-cli`
ever extends beyond personal scope. Even if Baasils (§3.1) ships v0.5
caries, Joseph annotating ~5-10 BWs/week in the background builds the
v1.0 fine-tune corpus.

## 5. Path F — Abandon caries entirely (REJECTED)

### The negative case (steelmanned)

Bone-loss alone is a meaningful chairside doc-aid:
- Periodontal assessment is recurring at every prophy visit; caries
  detection on BW is less frequent (every 6-18 months).
- Bone-loss measurement is more time-consuming to do by hand (CEJ → bone
  crest in mm) than caries detection (eyeball + tier label), so the
  automation lift per finding is higher.
- Caries findings can stay manual; bone-loss baseline-establishment is
  cumbersome enough that automation has real chair-time value.
- The "narrow vertical depth first" rule says: ship bone-loss tonight,
  see if it's load-bearing, then decide on caries.

### Why I still reject Path F

1. **The use case is a chairside doc-aid, not a periodontal-only tool.**
   A doc-aid that mute-skips caries when caries is one of the two most
   common BW findings will get rejected by the user (Joseph + wife) as
   incomplete. The aspirational form factor is "open BW → get all
   findings → edit → save." Half a tool is friction.

2. **Insurance narratives require depth.** Per the search of dental
   narrative requirements (Overjet, Dentistry IQ, Teero), the canonical
   narrative phrase is "tooth #X presents with [O/M/D] carious lesion
   extending into [enamel/dentin/pulp]." Without depth tiers, the
   findings JSON can't auto-fill this phrase.

3. **The work to keep caries is small.** Per §3.1, if Baasils probes
   clean, caries ships in ~2 hours. The cost-benefit doesn't favor
   abandoning a 2-hour task.

4. **Sunk-cost-aware reframing:** The 14 caries tests are already passing.
   `analyze.py` graceful-skip semantics already exist. The marginal work
   between v0 ship and v0.5 caries is data + train, not code. Code is
   the slow part. We're past it.

If hour-5 gate fails on bone-loss AND Baasils probe fails AND active
learning bootstrap fails, then Path F becomes the right call. Until then,
keep caries scoped in.

### Salvage value

The `graceful-skip` semantics already implement Path F as a runtime fallback.
If `weights/caries.pt` doesn't exist, the orchestrator emits empty
`caries_findings: []` and life goes on. v0 already ships in this mode.
Path F is therefore the **null hypothesis** that we revert to if everything
else fails — not a deliberate choice.

### 6.2 Active-learning bootstrap (HIGHEST-VALUE NEW IDEA)

Instead of hand-labeling 500 BWs from scratch (§4 = 17-25 hours),
combine §3.4 + §4:

1. Train a **binary caries detector** on DentalAI BW subset (~1,000
   images, single `Caries` class, CC-BY 4.0).
2. Run that binary model over Joseph's pool of ~500 unannotated BWs to
   pre-place bounding boxes.
3. Joseph reviews/corrects bounding boxes AND assigns ICCMS tier
   (RA1...RC6). Cost: 10-20 sec per pre-placed box, far less than
   placing from scratch.
4. **Target: 150 reviewed BWs** as the v0.5 fine-tune set, fine-tuning
   from the binary detector.

Time budget: ~3-4 hours of clinical-time annotation. v0.5 ships with
depth tiers from Joseph's own office BWs.

**Why this beats Path E alone:** annotation is 4-5x faster when the
geometry is pre-placed; Joseph only judges depth tier on lesions the
model already found. The cost is a low-recall floor (model misses lesions
→ Joseph manually scans for missed ones), but on a 150-BW corpus that's
manageable.

**Why this beats §3.1 alone:** even if Baasils probes clean, this is the
v1.0 trajectory anyway. Doing it in parallel costs nothing and starts
the office-specific compounding.

**Recommended actually-do-both posture:** Baasils ships v0.5 fast;
active-learning starts the v1.0 fine-tune corpus.

### 6.3 Synthetic generation via diffusion

Reviewed — well-studied in 2024-2025 (DentoMorph-LDM, DentoSMART, GAN-based
OPG pipelines). One paper reports "31.7% accuracy improvement for
early-stage caries" via synthetic augmentation.

**Why I don't recommend for v0.5:** To train a diffusion model that produces
realistic caries-bearing BWs, you need a seed dataset of real
caries-bearing BWs. Same chicken-and-egg as the original problem. The
technique is right for v1.0 once Joseph has ~200 real ICCMS-labeled BWs
to seed the LDM; it's not a v0.5 fast path.

### 6.4 Commercial APIs (REJECTED)

Pearl, Overjet, VideaHealth all offer dental caries detection APIs. Even
the tier-1 free trials would expose Joseph's BWs to a third party
(HIPAA-adjacent concern even for personal use) and would not produce
redistributable weights. Their outputs are also typically binary
present/absent with confidence scores — not ICCMS tiers. Reject.

### 6.5 Open-source dental ML communities

There are active Slack/Discord communities (e.g. r/Dentistry has an
AI sub-thread; "Dental AI Research" Discord; the OHDSI dental community)
but none are appropriate channels to ask for labeled BW data. Reject for
v0.5; useful for v1.0 model-validation collaborators.

### 6.6 Federated learning with another practice

If Joseph's wife's practice is the SAME office (per project context),
this is already covered. If there's a second practice in the network,
running training across two PHI-isolated sites would multiply the data
without consolidating it. Not relevant at current scope; revisit for
v1.0+.

## 8. Recommendation

### Primary action (do this first, takes ≤1 hour to know)

**Probe Baasils** with the existing probe scripts. Verify:
- `names:` field is `[Negative, Positive, RA1, RA2, RA3, RB4, RC5, RC6]` exactly
- Per-class real annotation counts roughly match Salehizeinabadi 2025's
  reported model performance (RC6 ≥ 30)
- Sample images are bitewing, not panoramic/photo

### If probe passes (high-probability outcome)

1. Patch `caries_adapter.py` to:
   - Drop `Negative` and `Positive` classes during YOLO export
   - Keep the existing 3-class collapse (initial/moderate/deep)
2. Run `scripts/download_caries.sh` retargeted at Baasils
3. Run `scripts/train_caries.sh` on pickles
4. Drop weights at `weights/caries.pt`; graceful-skip flips automatically
5. **v0.5 ships with real ICCMS-tier depth detection.**

### If probe fails

1. Spin up LabelStudio in Docker on pickles
2. Train a binary caries detector on the BW subset of DentalAI (CC-BY 4.0)
3. Use it as Label Studio's ML backend for pre-annotation
4. Joseph labels 150 of his own BWs in 3-4 hours over a long weekend,
   confirming/editing pre-placed boxes and assigning ICCMS tier
5. Fine-tune from the binary checkpoint to produce the 3-class model
6. In parallel, email Salehizeinabadi for the underlying dataset
7. v0.5 ships in 5-7 days, with a compounding office-specific corpus
   that improves at v0.6, v0.7, …

### Things explicitly NOT to do

- Don't burn 8 hours cleaning Renielaz — math doesn't change (Path B)
- Don't abandon caries entirely — chairside doc-aid needs both findings (Path F)
- Don't use HUNT4 / AI-Dentify — access timeline is months, not weeks
- Don't pay for Roboflow Pro to use Modify Classes — Baasils probe is free
  and likely solves the whole problem
- Don't synthetic-generate yet — it's a v1.0 augmentation strategy, not a
  v0.5 cold-start
